# 냉털 앱 보안/성능/PWA 코드 리뷰 (2026-04-15)

## TL;DR (우선순위)
1. **치명적**: Firebase RTDB 규칙이 사실상 전체 쓰기 허용 상태입니다.
2. **치명적**: 커뮤니티 데이터가 `innerHTML` 문자열 결합으로 렌더링되어 XSS 주입 위험이 큽니다.
3. **높음**: 관리자 인증이 클라이언트 해시 비교 + `localStorage` 플래그라 우회가 쉽습니다.
4. **중간**: `index.html` 단일 초대형(약 432KB, 5412줄) 구조로 초기 파싱/실행 비용이 큽니다.
5. **중간**: 문서 종료 태그(`</html>`) 뒤에 `<script>`가 남아 있어 브라우저 파서 의존 동작입니다.

## 세부 진단

### 1) 보안
- **Firebase Rules**
  - 현재 `.read: true`이고, 주요 노드들이 대부분 `.write: true`입니다.
  - 인증/검증 조건이 없어 누구나 임의 데이터 덮어쓰기·삭제·스팸 쓰기가 가능합니다.
- **XSS**
  - `esc()`는 `'`만 이스케이프하고 HTML 특수문자를 인코딩하지 않습니다.
  - 커뮤니티 텍스트/닉네임/레시피명 등이 HTML 문자열로 직접 합쳐져 렌더링됩니다.
- **관리자 인증**
  - 비밀번호 해시가 프런트 코드에 고정되어 있고, 통과 시 `localStorage.nt_admin='1'`을 저장합니다.
  - DevTools에서 `localStorage.setItem('nt_admin','1')`만으로 관리자 UI 접근이 가능합니다.
- **Firebase API Key 노출**
  - Firebase 웹 API key 노출 자체는 일반적이지만, 현재처럼 DB 규칙이 느슨하면 실질 공격면이 확대됩니다.

### 2) 성능/구조
- `index.html`이 매우 커서(432,336 bytes) 브라우저의 파싱/컴파일/실행 부담이 큽니다.
- 다수의 `render()` 재호출 + 전체 `innerHTML` 재생성이 반복되어 런타임 비용이 증가할 수 있습니다.
- 모듈 분리(데이터, 렌더러, Firebase, PWA), 탭 단위 지연 로딩, 이벤트 위임이 필요합니다.

### 3) PWA
- 장점: SW 설치/활성화/캐시 정리 로직이 있고, `beforeinstallprompt` 처리도 존재합니다.
- 개선점:
  - manifest에 `id`, `scope` 추가 권장.
  - iOS는 `beforeinstallprompt` 미지원이므로 현재 버튼 UX 안내 문구를 iOS 전용으로 분기하는 것이 좋습니다.
  - `</html>` 이후 스크립트는 제거/이동 필요.

### 4) 코드 품질/유지보수
- 인라인 스타일/인라인 핸들러(`onclick`)가 매우 많아 변경 파급 범위가 큽니다.
- 문자열 템플릿 기반 UI 조립이 커지며, 타입/경계조건 검증이 약해집니다.

### 5) 앱 디자인/UI 추천 개선 방향 (효과 큰 순서)
1. **터치 영역 확대 (가장 즉효)**
   - 하단 탭, 작은 버튼을 최소 44px 이상으로 맞추면 오터치가 크게 줄어듭니다.
   - 특히 한 손 조작이 많은 앱이라 체감 효과가 큽니다.
2. **포커스 표시 추가 (접근성 + 사용성)**
   - 키보드/보조기기 사용 시 현재 선택된 버튼이 보이도록 `:focus-visible` 스타일을 넣습니다.
   - "어디가 선택됐는지"가 명확해져 초보 사용자도 덜 헷갈립니다.
3. **문구 톤 통일 (불안 문구 줄이기)**
   - 상태 메시지는 경고 중심보다 "현재 무엇을 하는지"를 먼저 보여줍니다.
   - 예: "오류" 단독 표기보다 "잠시 후 자동 재시도" 같이 다음 행동을 안내.
4. **탭/필터 시각 우선순위 정리**
   - 선택된 탭 대비를 더 키우고, 보조 정보(예: 카운트)는 한 톤 낮춰 핵심 정보 집중도를 높입니다.
5. **로딩/빈 화면 UX 표준화**
   - 로딩 중/데이터 없음/오류 상태를 각각 다른 문구+아이콘으로 고정하면 학습 비용이 줄어듭니다.

## 즉시 실행 권고 (실행 순서)
1. Firebase Rules 최소권한화 (auth + validate 도입).
2. 커뮤니티 렌더링 `textContent`/DOM API 전환 + sanitizer 도입.
3. 관리자 기능을 서버 검증(Cloud Functions/Apps Script 서명 토큰)으로 이전.
4. `index.html`을 최소 3개 모듈로 분리(앱 셸/커뮤니티/관리자).
5. 문서 종료 후 스크립트 정리 및 lint 도입.
