# 생성형 이미지 앱 적용 절차 (초보자용)

> 목표: "이미지 생성 → 파일 저장 → 앱 연결 → 검증"을 안전하게 한 번에 끝내기

## 0) 먼저 정할 것 (5분)

1. 어떤 화면을 바꿀지 1개만 고르기
   - 예: 커뮤니티 기본 썸네일
2. 이미지 스타일 1개만 고정하기
   - 예: 따뜻한 파스텔, 둥근 느낌
3. 파일 규칙 정하기
   - 예: `assets/generated/community/thumb_default_01.webp`

---

## 1) 프롬프트 작성

아래 4요소를 꼭 넣으면 결과가 안정적입니다.

- **주제**: 무엇을 그릴지
- **스타일**: 플랫/실사/일러스트
- **색감**: 브랜드 컬러 또는 톤
- **용도**: 앱 썸네일/배경/아이콘

예시 프롬프트:

- `korean home-cooked dish thumbnail, soft pastel, mobile app UI, clean composition, no text`

팁:
- "no text", "no watermark"를 넣으면 실사용하기 쉬워요.

---

## 2) 이미지 생성 (3~6장)

- 한 장만 만들지 말고 3~6장 생성
- 가장 잘 나온 1~2장만 선택

선정 기준:
- 작은 화면에서 잘 보이는지
- 색 대비가 충분한지
- 너무 복잡하지 않은지

---

## 3) 후처리 (필수)

1. 크기 통일
   - 썸네일: 정사각형(예: 1024x1024)
2. 용량 줄이기
   - 가능하면 `webp` 사용
3. 안전 영역 확인
   - 모서리 잘림 없는지

권장 용량:
- 카드 썸네일 1장당 100~250KB

---

## 4) 저장 위치/이름 규칙

- 예시 폴더: `assets/generated/community/`
- 파일명 규칙 예시:
  - `thumb_default_01.webp`
  - `thumb_default_02.webp`
  - `thumb_default_03.webp`

규칙을 고정하면 나중에 교체가 쉬워요.

---

## 5) 앱 코드 연결

현재 구조에서는 `app-community.js`의 기본 썸네일 로직을 활용하면 됩니다.

연결 방법 A (가장 쉬움)
- 기존 SVG fallback 유지
- 특정 조건일 때만 로컬 생성 이미지 파일(`assets/generated/...`) 사용

연결 방법 B (디자인 우선)
- 사진이 없으면 항상 생성 이미지 파일 사용
- 이미지 실패 시 SVG fallback으로 2차 폴백

핵심 원칙:
- "이미지 실패해도 화면이 비지 않게" 폴백을 남겨두세요.

---

## 6) 실제 확인 체크리스트

- 사진 있는 글: 기존 사진 잘 보임
- 사진 없는 글: 생성 이미지 보임
- 깨진 URL 글: 자동 폴백 보임
- 스크롤 시 버벅임 없는지 확인
- 다크/라이트 테마에서 글자 대비 확인

---

## 7) 배포 전 최소 테스트

- 문법 체크: JS 에러 없는지
- 스모크 체크: 주요 화면 동작 확인
- 모바일 실기기 1회 확인

---

## 8) 추천 진행 순서 (실전)

1. 커뮤니티 기본 썸네일 3장 생성
2. 앱에 1장만 먼저 연결
3. 문제 없으면 3장 로테이션 적용
4. 이후 레시피 히어로/아이콘으로 확장

---

## 빠른 요약

- 한 번에 크게 바꾸지 말고, **화면 1개**부터
- 생성 이미지는 반드시 **용량 최적화 + 폴백 유지**
- 먼저 커뮤니티 썸네일부터 시작하는 게 가장 안전
